## Welcome Contributors!

Thank you for your interest in contributing to Mindect! Mindect provides documentation for everyone who wants to learn about AI, ML, Supervised Learning, and Advanced Learning Algorithms.

We welcome contributions from the community to help make our documentation better, more accurate, and more accessible.

## Getting Started

<Steps>
<Step>
### Clone the Repository

First, clone the Mindect repository to your local machine:

```bash title="terminal"
git clone https://github.com/yourusername/mindect
cd mindect
```

</Step>
<Step>
### Install Dependencies

Install all required dependencies using npm:

```bash title="terminal"
npm install
```

</Step>
<Step>
### Start the Development Server

Run the development server to preview your changes:

```bash title="terminal"
npm run dev
```

The site will be available at `http://localhost:3000`.

</Step>
</Steps>

## Contribution Workflow

<Steps>
<Step>
### Fork and Create a Branch

Create a fork of the repository and create a new branch for your changes:

```bash title="terminal"
git checkout -b feature/your-feature-name
```

Or for bug fixes:

```bash title="terminal"
git checkout -b fix/your-bug-fix
```

</Step>
<Step>
### Make Your Changes

- Follow the existing code style and documentation format
- Test your changes locally using `npm run dev`
- Ensure all links and references work correctly
- Add appropriate frontmatter to new MDX files

</Step>
<Step>
### Commit Your Changes

Write clear, descriptive commit messages:

```bash title="terminal"
git add .
git commit -m "Add: explanation of gradient descent optimization"
```

Commit message conventions:
- `Add:` for new features or content
- `Fix:` for bug fixes
- `Update:` for improvements to existing content
- `Docs:` for documentation-only changes

</Step>
<Step>
### Push and Create a Pull Request

Push your changes and create a pull request:

```bash title="terminal"
git push origin feature/your-feature-name
```

Then open a pull request on GitHub with:
- A clear title describing your changes
- A detailed description of what you've added or fixed
- Any relevant issue numbers (e.g., "Fixes #123")
- Screenshots if you've made visual changes

</Step>
</Steps>

## Content Guidelines

### Writing Documentation

- Use clear, concise language
- Explain concepts in simple terms
- Provide code examples where appropriate
- Include visual aids (diagrams, charts) when helpful
- Break down complex topics into smaller sections

### MDX Components

Mindect uses various MDX components. Here are some examples:

#### Steps Component

```mdx
<Steps>
<Step>
### Step Title
Step content here
</Step>
</Steps>
```

#### Callout Component

```mdx
<Callout type="info">
  Important information for readers
</Callout>
```

Types: `info`, `warning`, `error`, `success`

#### Code Blocks

````mdx
```python title="example.py"
def hello_world():
    print("Hello, World!")
```
````

#### Cards Component

```mdx
<Card href="/path" title="Card Title" />
```

### Frontmatter Format

All MDX files should include frontmatter:

```mdx
---
title: Page Title
description: Brief description of the page content
---
```

## Code Style

- Use consistent indentation (2 spaces for JavaScript/MDX)
- Follow existing naming conventions
- Comment complex code sections
- Keep functions small and focused
- Use TypeScript types where applicable

## Testing Your Changes

Before submitting a pull request:

1. **Build the project** to ensure no errors:
   ```bash title="terminal"
   npm run build
   ```

2. **Test locally** using the development server:
   ```bash title="terminal"
   npm run dev
   ```

3. **Check for broken links** and verify all navigation works

4. **Review your content** for typos and formatting issues

## Jupyter Notebooks

When contributing Jupyter notebooks:

- Clear all outputs before committing (to reduce file size)
- Include markdown cells with explanations
- Use descriptive variable names
- Add comments to complex code
- Ensure notebooks run from top to bottom without errors
- Include any required data files or provide download links

## Reporting Issues

Found a bug or have a suggestion? Please open an issue:

### Bug Reports

Include:
- Clear description of the bug
- Steps to reproduce
- Expected vs actual behavior
- Screenshots if applicable
- Browser/OS information

### Feature Requests

Include:
- Clear description of the proposed feature
- Use cases and benefits
- Any relevant examples or mockups
